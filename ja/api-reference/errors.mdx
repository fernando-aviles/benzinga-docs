---
title: "エラー"
description: "Benzinga API のエラーとステータスコードへの対処方法"
---

Benzinga API では、API リクエストの成功または失敗を示すために、標準的な HTTP レスポンスコードを使用します。一般的には、次のとおりです。

* **2xx**: 成功。
* **4xx**: クライアントエラー (例: パラメータ不足、無効なキー) 。
* **5xx**: サーバーエラー (Benzinga 側で問題が発生したことを示します) 。

> [!TIP]
> **ベストプラクティス**: リクエストが成功したかどうかを判断するには、まず **HTTP ステータスコード** を確認してください。API エンドポイントによって形式が異なる場合があるため、レスポンス本文だけに頼らないでください。

<div id="http-status-codes">
  ## HTTPステータスコード
</div>

次の表は、一般的なステータスコードの一覧です。

| コード     | ステータス                   | 説明                                                                   |
| :------ | :---------------------- | :------------------------------------------------------------------- |
| **200** | `OK`                    | リクエストは成功しました。                                                        |
| **400** | `Bad Request`           | リクエストは受け付けられませんでした。通常は、パラメータの不足または無効な値が原因です。                         |
| **401** | `Unauthorized`          | 有効なAPI Keyが指定されていません。`Authorization` ヘッダーまたは `token` パラメータを確認してください。 |
| **402** | `Request Failed`        | パラメータは有効でしたが、ビジネスロジック上の理由によりリクエストは失敗しました。                            |
| **403** | `Forbidden`             | API Keyは有効ですが、このリソースにアクセスする権限がありません。                                 |
| **404** | `Not Found`             | 指定されたリソース (例: ID、エンドポイント) は存在しません。                                   |
| **429** | `Too Many Requests`     | レート制限を超えています。                                                        |
| **500** | `Internal Server Error` | Benzingaのサーバー側で問題が発生しました。これはまれです。                                    |
| **503** | `Service Unavailable`   | サービスは一時的に利用できません (例: メンテナンス中) 。                                      |

<div id="error-response-bodies">
  ## エラーレスポンスボディ
</div>

HTTPステータスコードはエラーを示す主な指標ですが、レスポンスボディにもデバッグに役立つ詳細情報が含まれていることがよくあります。このボディの形式は、呼び出すAPIによって異なります。

<div id="format-1-simple-error-message">
  ### 形式 1: シンプルなエラーメッセージ
</div>

多くのエンドポイント (ニュース API など) では、メッセージを含むシンプルな文字列、またはフラットな JSON オブジェクトが返されます。

```json
"Invalid or Missing Query Parameters"
```

または

```json
{
  "message": "Invalid page size"
}
```

<div id="format-2-structured-error-object">
  ### 形式 2: 構造化エラーオブジェクト
</div>

新しい API (Data API Proxy / ファンダメンタルズ など) では、エラーの一覧を含む構造化オブジェクトが返されます。

```json
{
  "data": null,
  "errors": [
    {
      "code": "database_query_error",
      "id": "ERR_12345",
      "value": "Unable to fetch data for symbol: INVALID"
    }
  ],
  "ok": false
}
```

<div id="handling-errors-programmatically">
  ### プログラムでエラーを処理する
</div>

レスポンスボディの内容は一定でない可能性があるため、堅牢なエラー処理戦略を推奨します。

1. **HTTPステータスコードを確認します。** `>= 400` の場合は、エラーとして扱います。
2. **レスポンスボディをログに記録します。** デバッグのため、ボディ全体をログに出力します。
3. **汎用的なメッセージを表示します。** 特定のエンドポイントに連携しており、その正確なエラー形式を把握している場合を除き、エンドユーザーにはステータスコードとあわせて「問題が発生しました」という汎用的なメッセージを表示します。
