---
title: "エラー"
description: "Benzinga API のエラーとステータスコードの扱い方"
---

Benzinga API では、API リクエストの成功または失敗を示すために、一般的な HTTP レスポンスコードを使用します。概要は次のとおりです。

* **2xx**: 成功。
* **4xx**: クライアントエラー (例: パラメータの不足、無効なキー) 。
* **5xx**: サーバーエラー (Benzinga 側で問題が発生した場合) 。

> [!TIP]
> **ベストプラクティス**: リクエストが成功したかどうかは、必ず最初に **HTTPステータスコード** を確認して判断してください。レスポンスの形式は API エンドポイントによって異なる場合があるため、レスポンスボディだけに依存しないでください。

<div id="http-status-codes">
  ## HTTPステータスコード
</div>

次の表に、一般的なステータスコードを示します。

| コード     | ステータス                   | 説明                                                                 |
| :------ | :---------------------- | :----------------------------------------------------------------- |
| **200** | `OK`                    | リクエストは成功しました。                                                      |
| **400** | `Bad Request`           | リクエストは受け付けられませんでした。多くの場合、パラメータの不足または無効な値が原因です。                     |
| **401** | `Unauthorized`          | 有効なAPIキーが指定されていません。`Authorization` ヘッダーまたは `token` パラメータを確認してください。 |
| **402** | `Request Failed`        | パラメータは有効でしたが、ビジネスロジック上の理由でリクエストは失敗しました。                            |
| **403** | `Forbidden`             | APIキーは有効ですが、このリソースにアクセスする権限がありません。                                 |
| **404** | `Not Found`             | 指定されたリソース (例: ID、エンドポイント) は存在しません。                                 |
| **429** | `Too Many Requests`     | レート制限を超過しました。                                                      |
| **500** | `Internal Server Error` | Benzinga のサーバー側で問題が発生しました。これはまれです。                                 |
| **503** | `Service Unavailable`   | サービスは一時的に利用できません (例: メンテナンス中) 。                                    |

<div id="error-response-bodies">
  ## エラーレスポンスボディ
</div>

HTTPステータスコードはエラーを示す主要な指標ですが、レスポンスボディにもデバッグに役立つ詳細情報が含まれていることがよくあります。このボディの形式は、呼び出すAPIによって異なる場合があります。

<div id="format-1-simple-error-message">
  ### 形式 1: シンプルなエラーメッセージ
</div>

多くのエンドポイント (News API など) では、シンプルな文字列、またはメッセージを含む単純な JSON オブジェクトが返されます。

```json
"Invalid or Missing Query Parameters"
```

または

```json
{
  "message": "Invalid page size"
}
```

<div id="format-2-structured-error-object">
  ### 形式 2: 構造化されたエラーオブジェクト
</div>

新しい API (Data API Proxy / ファンダメンタルズ など) では、エラーのリストを含む構造化オブジェクトが返されます。

```json
{
  "data": null,
  "errors": [
    {
      "code": "database_query_error",
      "id": "ERR_12345",
      "value": "Unable to fetch data for symbol: INVALID"
    }
  ],
  "ok": false
}
```

<div id="handling-errors-programmatically">
  ### エラーをプログラムで処理する
</div>

レスポンスボディは一定ではない可能性があるため、堅牢なエラー処理戦略を推奨します。

1. **HTTPステータスコードを確認します。** `>= 400` の場合は、エラーとして扱います。
2. **レスポンスボディをログに記録します。** デバッグのため、ボディ全体をログに出力します。
3. **汎用的なメッセージを表示します。** 特定のエンドポイントと連携していて、その正確なエラー形式を把握している場合を除き、ステータスコードとあわせてエンドユーザーには「問題が発生しました」のような汎用的なメッセージを表示します。
