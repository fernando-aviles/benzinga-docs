<div id="overview">
  ## 概要
</div>

このエンドポイントを使用すると、設定済みのエンドポイントにテスト用の webhook 配信をトリガーできます。これにより、本番環境のライブデータを受信する前に、webhook 連携が正しく機能しているかを確認できます。

<div id="testing-your-integration">
  ## 連携のテスト
</div>

このエンドポイントを呼び出すと、Benzinga は設定した `destination` URL にテスト用のWebhookペイロードを送信します。このテスト配信でも、本番環境のWebhook配信と同じ形式および再試行ロジックが使用されます。

<div id="what-to-expect">
  ### 想定される結果
</div>

1. **即時レスポンス**: テスト配信が正常にトリガーされると、API は `200` ステータスコードを返します
2. **テストペイロード**: Webhook エンドポイントは、本番データと同じ形式のテストペイロードを受信します
3. **配信ヘッダー**: テスト配信には、本番環境への配信と同様に `X-BZ-Delivery` ヘッダーが含まれます

<div id="verify-your-integration">
  ### 連携を検証する
</div>

このエンドポイントを使用して、次の点を確認してください。

* Webhook エンドポイントが外部からアクセス可能であること
* エンドポイントが Webhook ペイロードの形式を正しく解析できること
* エンドポイントが適切なステータスコード (成功時は 2xx) を返すこと
* エンドポイントが 30 秒のタイムアウト以内に応答すること
* 冪等性ロジックが `X-BZ-Delivery` ヘッダーとペイロードの `ID` フィールドを正しく処理できること

<div id="best-practices">
  ## ベストプラクティス
</div>

* まずは非本番の webhook エンドポイントでテストしてください
* エンドポイントが `200` または `204` のステータスコードを返すことを確認してください
* ログと監視でテスト配信を確実に捕捉できることを確認してください
* 重複排除ロジックがテスト配信 ID で正しく機能することを確認してください
* 一時的にエラーステータスコードを返して、エラーシナリオをテストしてください

<div id="troubleshooting">
  ## トラブルシューティング
</div>

<div id="424-delivery-error">
  ### 424 配信エラー
</div>

`424` ステータスコードを受け取った場合、システムがテストペイロードを送信先エンドポイントに配信できなかったことを示します。一般的な原因は次のとおりです。

* 送信先 URL にパブリックアクセスできない
* 送信先エンドポイントがエラーのステータスコードを返している
* ネットワーク接続の問題
* 送信先エンドポイントの SSL/TLS 証明書エラー

<div id="400-invalid-request">
  ### 400 無効なリクエスト
</div>

必須のパラメータがすべて指定されており、形式も正しいことを確認してください。

* `destination` は有効な HTTPS URL である必要があります
* `version` は `webhook/v1` である必要があります
* `kind` は `News/v1` である必要があります
