Webhookエンジンは、再試行、フィルタリング、変換のオプションを備え、カレンダーとシグナルの更新をHTTPエンドポイントに直接プッシュ配信します。配信の仕組み、ペイロード形式、統合のベストプラクティスを理解するために、このガイドを参照してください。

<div id="overview">
  ## 概要
</div>

Benzinga Data Webhook サービスは、リアルタイムのカレンダーおよびシグナルデータを、設定した webhook エンドポイントに配信します。カレンダーイベント (決算、配当、レーティングなど) またはシグナル (オプションアクティビティ、取引停止など) が作成、更新、または削除されると、このサービスはデータペイロードを含む HTTP POST リクエストを webhook URL に自動的に送信します。

主な機能:

* カレンダーおよびシグナルの配信範囲を設定できるため、必要なデータのみを受信可能
* 重複排除のための一意の `X-BZ-Delivery` ヘッダーとペイロードの `id` フィールドを備えた冪等な配信
* 高速な指数バックオフによる再試行から、長時間にわたる 1 時間ごとの再試行へと移行する堅牢な再試行スケジュール
* ペイロードを下流システムの要件に合わせるためのオプションのコンテンツ変換

<div id="webhook-delivery">
  ## Webhookの配信
</div>

<div id="http-request-details">
  ### HTTP リクエストの詳細
</div>

* **Method**: `POST`
* **Content-Type**: `application/json`
* **User-Agent**: `Benzinga-Dispatch/v1.0.0 {build-version}`
* **Custom Header**: `X-BZ-Delivery` - 配信の試行ごとに一意の UUID (重複排除に役立ちます)

<div id="retry-policy">
  ### 再試行ポリシー
</div>

データWebhookサービスでは、堅牢な再試行メカニズムを実装しています。

* **指数バックオフフェーズ**: 最初の5分間で15回再試行
* **追加の指数バックオフ再試行**: 必要に応じてさらに11回再試行
* **固定間隔フェーズ**: 1時間あたり12回の再試行 × 24時間/日 × 7日間 (長期的な再試行向け) 
* **最大待機時間**: 指数バックオフフェーズでの再試行間隔は最大5分
* **リクエストタイムアウト**: 各リクエストにつき30秒

<div id="response-requirements">
  ### レスポンス要件
</div>

Webhook エンドポイントは、次のいずれかの HTTP ステータスコードを返す必要があります。

* **成功コード** (200-202, 204): 配信が正常に完了したことを示します。再試行は行われません。
* **クライアントエラーコード** (401-403): 認証または認可の失敗を示します。以降の失敗を防ぐため、再試行は**直ちに停止**されます。
* **その他のコード** (4xx, 5xx): 上記の再試行ポリシーに従って再試行されます。

**重要**: タイムアウトを避けるため、エンドポイントは速やかに (理想的には 30 秒以内に) 応答してください。タイムアウトした場合、エンジンは再試行します。

<div id="payload-fields">
  ### ペイロードのフィールド
</div>

<div id="top-level-fields">
  #### 最上位フィールド
</div>

* **`id`** (string, UUID): この Webhook 配信の一意の識別子です。重複排除に使用します。
* **`api_version`** (string): API バージョン識別子です。現在は `"webhook/v1"` です。
* **`kind`** (string): データ種別のパス識別子です。指定可能な値については、[対応データ種別](#supported-data-types)を参照してください。

<div id="data-object">
  #### Data Object
</div>

* **`action`** (string): イベントのアクション種別。指定可能な値:
  * `"Created"`: 新しいデータが作成されたことを示します (新しい webhook キーのデフォルト値) 
  * `"Updated"`: 既存のデータが更新されたことを示します
  * `"Removed"`: データが削除されたことを示します
  * **Note**: レガシー Webhook キーでは、小文字の値を受け取る場合があります: `"created"`, `"updated"`, `"removed"`
* **`id`** (string): カレンダー/シグナル レコードの一意の識別子
* **`timestamp`** (string, ISO 8601): webhook が生成された日時のタイムスタンプ
* **`content`** (object): 実際の カレンダー または シグナル のデータ。構造はデータ種別によって異なります ([サポートされるデータ種別](#supported-data-types) を参照)

<div id="event-actions">
  ## イベントアクション
</div>

データ Webhook サービスは、次の3種類のアクションに対応するイベントを送信します。

1. **Created**: 新しいカレンダーデータまたはシグナルデータが公開されたときに発生します
2. **Updated**: 既存のデータが更新されたときに発生します
3. **Removed**: データが削除されたときに発生します

**注**: アクションの形式は、Webhook の設定によって異なります。

* **新しい Webhook キー**: 先頭が大文字のアクションを受信します (`"Created"`、`"Updated"`、`"Removed"`) 
* **レガシー Webhook キー**: 小文字のアクションを受信します (`"created"`、`"updated"`、`"removed"`)

<div id="content-filtering">
  ## コンテンツのフィルタリング
</div>

Webhook の設定には、受信するデータを制御するためのフィルターを含めることができます。

<div id="filter-options">
  ### フィルターオプション
</div>

* **データ種別**: 特定のカレンダー/シグナル種別でフィルタリングします (例: 決算のみ、レーティングのみ) 
* **地理的フィルター**: 受信するデータを制御します:
  * 米国市場データ (`AllowUSA`)
  * カナダ市場データ (`AllowCanada`)
  * インド市場データ (`AllowIndia`) - WIIMs データ向け
* **日付フィルター**: 指定した日付より古い履歴データを除外します (`MaxHistoricalDate`)

<div id="exchange-filtering">
  ### 取引所によるフィルタリング
</div>

このサービスでは、地域設定に応じて取引所が自動的に絞り込まれます。

* **米国の取引所**: NYSE, NASDAQ, AMEX, ARCA, OTC, OTCBB, PINX, PINK, BATS, IEX
* **カナダの取引所**: TSX, TSXV, CSE, CNSX

<div id="content-transformation">
  ## コンテンツ変換
</div>

Webhook エンジンは、特定のデータ型に対するコンテンツ変換をサポートしています。変換は webhook の設定に基づいて適用され、以下の処理が含まれる場合があります。

* フィールド名の変更
* データ形式の変換
* フィールドのフィルタリング/削除

<div id="best-practices">
  ## ベストプラクティス
</div>

<div id="1-idempotency">
  ### 1. 冪等性
</div>

ペイロード内の`id`フィールド (UUID) を使用して冪等性を実装します。重複処理を防ぐため、処理済みの配信IDを保存してください。

<div id="2-response-handling">
  ### 2. レスポンスの処理
</div>

* webhookを受信したら、直ちに `200 OK` または `204 No Content` を返してください
* 必要に応じて、dataは非同期で処理してください
* レスポンスを返す前に、時間のかかる処理は行わないでください

<div id="3-error-handling">
  ### 3. エラー処理
</div>

* 適切なHTTPステータスコードを返す
* 認証エラー (401～403) の場合は、エンドポイントが正しく設定されていることを確認する
* 一時的な障害については、再試行を発生させるために5xxステータスコードを返す

<div id="4-security">
  ### 4. セキュリティ
</div>

* webhook エンドポイントには HTTPS を使用してください
* 認証/認可 (API キー、token など) を実装してください
* セキュリティ強化のため、`X-BZ-Delivery` ヘッダーを検証してください

<div id="5-monitoring">
  ### 5. 監視
</div>

* エンドポイントのレスポンス時間を監視する
* 繰り返し発生する失敗に対するアラートを設定する
* 配信試行を識別するために `X-BZ-Delivery` ヘッダーを追跡する

<div id="6-data-type-handling">
  ### 6. データ種別の扱い
</div>

* `kind` フィールドを確認してデータ種別を判別します
* データ種別の構造に応じて `content` オブジェクトを解析します
* レガシーキーをサポートする場合は、action の形式の違い (先頭大文字か小文字か) に対応します

<div id="example-webhook-handler">
  ## Webhookハンドラーの例
</div>

<div id="python-flask">
  ### Python (Flask)
</div>

```python
from flask import Flask, request, jsonify
import uuid

app = Flask(__name__)
processed_ids = set()

@app.route('/webhook', methods=['POST'])
def webhook():
    # 重複排除用のデリバリーIDを取得
    delivery_id = request.headers.get('X-BZ-Delivery')
    
    # ペイロードを解析
    payload = request.json
    content_id = payload['id']
    
    # 重複チェック
    if content_id in processed_ids:
        return jsonify({'status': 'duplicate'}), 200
    
    # データを処理
    action = payload['data']['action']
    kind = payload['kind']
    content = payload['data']['content']
    
    print(f"Received {action} event for {kind}")
    print(f"Data ID: {content.get('id')}")
    
    # データ種別ごとの処理
    if 'earnings' in kind:
        print(f"Earnings: {content.get('ticker')} - {content.get('date')}")
    elif 'ratings' in kind:
        print(f"Rating: {content.get('ticker')} - {content.get('rating_current')}")
    elif 'option_activity' in kind:
        print(f"Option Activity: {content.get('ticker')} - {content.get('volume')}")
    
    # 処理済みとしてマーク
    processed_ids.add(content_id)
    
    # 即座に成功レスポンスを返す
    return jsonify({'status': 'received'}), 200
```

<div id="nodejs-express">
  ### Node.js (Express)
</div>

```javascript
const express = require('express');
const app = express();
app.use(express.json());

const processedIds = new Set();

app.post('/webhook', (req, res) => {
  // 重複排除用のデリバリーIDを取得
  const deliveryId = req.headers['x-bz-delivery'];
  
  // ペイロードを解析
  const payload = req.body;
  const contentId = payload.id;
  
  // 重複チェック
  if (processedIds.has(contentId)) {
    return res.status(200).json({ status: 'duplicate' });
  }
  
  // データを処理
  const { action, content } = payload.data;
  const kind = payload.kind;
  
  console.log(`Received ${action} event for ${kind}`);
  console.log(`Data ID: ${content.id}`);
  
  // データ種別ごとの処理
  if (kind.includes('earnings')) {
    console.log(`Earnings: ${content.ticker} - ${content.date}`);
  } else if (kind.includes('ratings')) {
    console.log(`Rating: ${content.ticker} - ${content.rating_current}`);
  } else if (kind.includes('option_activity')) {
    console.log(`Option Activity: ${content.ticker} - ${content.volume}`);
  }
  
  // 処理済みとしてマーク
  processedIds.add(contentId);
  
  // 即座に成功レスポンスを返す
  res.status(200).json({ status: 'received' });
});

app.listen(3000);
```

<div id="go">
  ### Go
</div>

```go
package main

import (
    "encoding/json"
    "fmt"
    "log"
    "net/http"
    "sync"
)

var (
    processedIDs = make(map[string]bool)
    mu           sync.RWMutex
)

type WebhookPayload struct {
    ID         string `json:"id"`
    APIVersion string `json:"api_version"`
    Kind       string `json:"kind"`
    Data       struct {
        Action    string                 `json:"action"`
        ID        string                 `json:"id"`
        Timestamp string                 `json:"timestamp"`
        Content   map[string]interface{} `json:"content"`
    } `json:"data"`
}

func webhookHandler(w http.ResponseWriter, r *http.Request) {
    // デリバリーIDを取得
    deliveryID := r.Header.Get("X-BZ-Delivery")
    
    // ペイロードを解析
    var payload WebhookPayload
    if err := json.NewDecoder(r.Body).Decode(&payload); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    
    // 重複チェック
    mu.RLock()
    if processedIDs[payload.ID] {
        mu.RUnlock()
        w.WriteHeader(http.StatusOK)
        json.NewEncoder(w).Encode(map[string]string{"status": "duplicate"})
        return
    }
    mu.RUnlock()
    
    // データを処理
    fmt.Printf("Received %s event for %s\n", payload.Data.Action, payload.Kind)
    fmt.Printf("Data ID: %s\n", payload.Data.ID)
    
    // データ種別ごとに処理
    content := payload.Data.Content
    if ticker, ok := content["ticker"].(string); ok {
        fmt.Printf("Ticker: %s\n", ticker)
    }
    
    // 処理済みとしてマーク
    mu.Lock()
    processedIDs[payload.ID] = true
    mu.Unlock()
    
    // 成功レスポンスを返す
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]string{"status": "received"})
}

func main() {
    http.HandleFunc("/webhook", webhookHandler)
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

</div>

<div id="common-issues">
  ### よくある問題
</div>

1. **Webhookを受信できない**
   * Webhook URL が正しく設定されていることを確認してください
   * エンドポイントが外部からアクセス可能であることを確認してください
   * API キーが有効かつアクティブであることを確認してください
   * フィルターですべてのデータ種別が除外されていないことを確認してください
   * 地理フィルターが想定しているデータと一致していることを確認してください

2. **重複配信**
   * `id` フィールドを使用して冪等性を実装してください
   * エンドポイントの応答時間を確認してください (応答が遅いと再試行が発生する場合があります) 

3. **認証エラー (401-403)**
   * エンドポイントの認証設定を確認してください
   * API キーと token を確認してください
   * 注: 認証エラーが発生すると、再試行は直ちに停止します

4. **タイムアウトエラー**
   * エンドポイントが 30 秒以内に応答することを確認してください
   * 必要に応じてデータを非同期で処理してください
   * すぐに成功レスポンスを返し、処理は後で実行してください

5. **想定外の action 形式**
   * レガシー Webhook キーを使用していないか確認してください (action が小文字) 
   * 先頭が大文字の action を受け取るには、新しい Webhook キーに更新してください
   * 複数のクライアントをサポートしている場合は、両方の形式に対応してください

6. **必要なデータ種別が届かない**
   * Webhook 設定に必要なデータ種別が含まれていることを確認してください
   * 地理フィルターを確認してください (US/Canada/India の設定) 
   * date フィルターによって最新のデータが除外されていないことを確認してください
