Webhook エンジンは、再試行、フィルタリング、変換の各オプションを備え、カレンダーとシグナルの更新を HTTP エンドポイントへ直接配信します。このガイドでは、配信の仕組み、ペイロードの形式、連携のベストプラクティスを確認できます。

<div id="overview">
  ## 概要
</div>

Benzinga Data Webhook サービスは、設定した webhook エンドポイントにリアルタイムのカレンダーデータとシグナルデータを配信します。カレンダーイベント (決算、配当、レーティング など) やシグナル (オプションアクティビティ、売買停止 など) が作成、更新、または削除されると、このサービスはデータペイロードを含む HTTP POST リクエストを webhook URL に自動的に送信します。

主な機能:

* カレンダーおよびシグナルの配信対象範囲を設定できるため、必要なデータのみを受信可能
* 重複排除のための一意な `X-BZ-Delivery` ヘッダーとペイロードの `id` フィールドを備えた冪等な配信
* 高速な指数バックオフによる再試行から、その後の長時間にわたる毎時の再試行へと移行する堅牢な再試行スケジュール
* 後続システムの要件に合わせてペイロードを調整するためのオプションのコンテンツ変換

<div id="webhook-delivery">
  ## Webhookの配信
</div>

<div id="http-request-details">
  ### HTTPリクエストの詳細
</div>

* **Method**: `POST`
* **Content-Type**: `application/json`
* **User-Agent**: `Benzinga-Dispatch/v1.0.0 {build-version}`
* **Custom Header**: `X-BZ-Delivery` - 配信試行ごとに一意のUUID (重複排除に役立ちます)

<div id="retry-policy">
  ### 再試行ポリシー
</div>

データWebhookサービスでは、堅牢な再試行メカニズムを実装しています。

* **指数バックオフフェーズ**: 最初の 5 分間で 15 回再試行
* **追加の指数バックオフ再試行**: 必要に応じてさらに 11 回再試行
* **固定間隔フェーズ**: 1 時間あたり 12 回の再試行 × 1 日 24 時間 × 7 日間 (長期的な再試行向け) 
* **最大待機時間**: 指数バックオフフェーズでの再試行間隔は最大 5 分
* **リクエストタイムアウト**: リクエストごとに 30 秒

<div id="response-requirements">
  ### レスポンス要件
</div>

Webhook エンドポイントは、以下のいずれかの HTTPステータスコード を返す必要があります。

* **成功コード** (200-202, 204): 配信が正常に完了したことを示します。再試行は行われません。
* **クライアントエラーコード** (401-403): 認証または認可の失敗を示します。以降の失敗を防ぐため、再試行は**直ちに停止**されます。
* **その他のコード** (4xx, 5xx): 上記の再試行ポリシーに従って再試行が行われます。

**重要**: タイムアウトを避けるため、エンドポイントは速やかに (理想的には 30 秒以内に) 応答する必要があります。タイムアウトした場合、エンジンは再試行を行います。

<div id="payload-fields">
  ### ペイロードのフィールド
</div>

<div id="top-level-fields">
  #### トップレベルのフィールド
</div>

* **`id`** (string, UUID): この webhook 配信の一意の識別子です。重複排除に使用します。
* **`api_version`** (string): API バージョンの識別子です。現在は `"webhook/v1"` です。
* **`kind`** (string): データ型パスの識別子です。使用可能なすべての値については、[サポートされているデータ型](#supported-data-types)を参照してください。

<div id="data-object">
  #### データオブジェクト
</div>

* **`action`** (string): イベントのアクション種別。指定可能な値:
  * `"Created"`: 新しいデータが作成されたことを示します (新しい Webhook キーのデフォルト値) 
  * `"Updated"`: 既存のデータが更新されたことを示します
  * `"Removed"`: データが削除されたことを示します
  * **注**: レガシーな Webhook キーでは、小文字の値を受け取る場合があります: `"created"`, `"updated"`, `"removed"`
* **`id`** (string): calendar/シグナル レコードの一意の識別子
* **`timestamp`** (string, ISO 8601): Webhook が生成された時点のタイムスタンプ
* **`content`** (object): 実際の calendar または シグナル のデータ。構造はデータ型によって異なります ([サポートされているデータ型](#supported-data-types) を参照)

<div id="event-actions">
  ## Event Actions
</div>

データWebhookサービスは、次の3種類のアクションに対するイベントを送信します。

1. **Created**: 新しいカレンダーデータまたはシグナルデータが公開されたときに発生します
2. **Updated**: 既存のデータが更新されたときに発生します
3. **Removed**: データが削除されたときに発生します

**注**: アクションの形式は、Webhookの設定によって異なります。

* **新しいWebhookキー**: 先頭が大文字のアクション (`"Created"`、`"Updated"`、`"Removed"`) を受信します
* **従来のWebhookキー**: 小文字のアクション (`"created"`、`"updated"`、`"removed"`) を受信します

<div id="content-filtering">
  ## コンテンツのフィルタリング
</div>

Webhook の設定には、受信するデータを制御するためのフィルタを含めることができます。

<div id="filter-options">
  ### フィルターオプション
</div>

* **データタイプ**: 特定のカレンダー/シグナルタイプで絞り込みます (例: 決算のみ、レーティングのみ) 
* **地域フィルター**: 次の市場データを受信するかどうかを制御します:
  * 米国市場データ (`AllowUSA`) 
  * カナダ市場データ (`AllowCanada`) 
  * インド市場データ (`AllowIndia`)  - WIIMs データ用
* **日付フィルター**: 指定した日付より前の履歴データを除外します (`MaxHistoricalDate`)

<div id="exchange-filtering">
  ### 取引所フィルタリング
</div>

このサービスでは、地域設定に応じて取引所によるフィルタリングが自動的に適用されます。

* **米国の取引所**: NYSE, NASDAQ, AMEX, ARCA, OTC, OTCBB, PINX, PINK, BATS, IEX
* **カナダの取引所**: TSX, TSXV, CSE, CNSX

<div id="content-transformation">
  ## コンテンツ変換
</div>

Webhook エンジンは、特定のデータ型に対するコンテンツ変換をサポートしています。変換は webhook の設定に基づいて適用され、次のような処理が含まれる場合があります。

* フィールド名の変更
* データ形式の変換
* フィールドの絞り込みまたは削除

<div id="best-practices">
  ## ベストプラクティス
</div>

<div id="1-idempotency">
  ### 1. 冪等性
</div>

冪等性を実装するには、ペイロード内の `id` フィールド (UUID) を使用します。重複処理を避けるため、処理済みの配信 ID を保存してください。

<div id="2-response-handling">
  ### 2. レスポンス処理
</div>

* webhookを受信したら、すぐに `200 OK` または `204 No Content` を返します
* 必要に応じて、データは非同期で処理します
* レスポンスを返す前に、時間のかかる処理は行わないでください

<div id="3-error-handling">
  ### 3. エラー処理
</div>

* 適切なHTTPステータスコードを返してください
* 認証エラー (401～403) の場合は、エンドポイントが正しく設定されていることを確認してください
* 一時的な障害が発生した場合は、再試行をトリガーするために5xxステータスコードを返してください

<div id="4-security">
  ### 4. セキュリティ
</div>

* webhookエンドポイントではHTTPSを使用します
* 認証/認可を実装します (APIキー、token など) 
* セキュリティ強化のため、`X-BZ-Delivery` ヘッダーを検証します

<div id="5-monitoring">
  ### 5. 監視
</div>

* エンドポイントのレスポンスタイムを監視する
* 繰り返し発生する障害に対するアラートを設定する
* 配信試行を識別するため、`X-BZ-Delivery` ヘッダーを追跡する

<div id="6-data-type-handling">
  ### 6. データ型の処理
</div>

* データ型を判別するには、`kind` フィールドを確認します
* データ型の構造に応じて `content` オブジェクトをパースします
* レガシーキーをサポートする場合は、異なるアクション形式 (先頭が大文字の形式と小文字の形式) に対応します

<div id="example-webhook-handler">
  ## Webhookハンドラーの例
</div>

<div id="python-flask">
  ### Python (Flask)
</div>

```python
from flask import Flask, request, jsonify
import uuid

app = Flask(__name__)
processed_ids = set()

@app.route('/webhook', methods=['POST'])
def webhook():
    # 重複排除用のデリバリーIDを取得
    delivery_id = request.headers.get('X-BZ-Delivery')
    
    # ペイロードを解析
    payload = request.json
    content_id = payload['id']
    
    # 重複チェック
    if content_id in processed_ids:
        return jsonify({'status': 'duplicate'}), 200
    
    # データを処理
    action = payload['data']['action']
    kind = payload['kind']
    content = payload['data']['content']
    
    print(f"Received {action} event for {kind}")
    print(f"Data ID: {content.get('id')}")
    
    # データ型に応じた処理
    if 'earnings' in kind:
        print(f"Earnings: {content.get('ticker')} - {content.get('date')}")
    elif 'ratings' in kind:
        print(f"Rating: {content.get('ticker')} - {content.get('rating_current')}")
    elif 'option_activity' in kind:
        print(f"Option Activity: {content.get('ticker')} - {content.get('volume')}")
    
    # 処理済みとしてマーク
    processed_ids.add(content_id)
    
    # 即座に成功レスポンスを返す
    return jsonify({'status': 'received'}), 200
```

<div id="nodejs-express">
  ### Node.js (Express)
</div>

```javascript
const express = require('express');
const app = express();
app.use(express.json());

const processedIds = new Set();

app.post('/webhook', (req, res) => {
  // 重複排除用のデリバリーIDを取得
  const deliveryId = req.headers['x-bz-delivery'];
  
  // ペイロードを解析
  const payload = req.body;
  const contentId = payload.id;
  
  // 重複チェック
  if (processedIds.has(contentId)) {
    return res.status(200).json({ status: 'duplicate' });
  }
  
  // データを処理
  const { action, content } = payload.data;
  const kind = payload.kind;
  
  console.log(`Received ${action} event for ${kind}`);
  console.log(`Data ID: ${content.id}`);
  
  // データ型ごとの処理
  if (kind.includes('earnings')) {
    console.log(`Earnings: ${content.ticker} - ${content.date}`);
  } else if (kind.includes('ratings')) {
    console.log(`Rating: ${content.ticker} - ${content.rating_current}`);
  } else if (kind.includes('option_activity')) {
    console.log(`Option Activity: ${content.ticker} - ${content.volume}`);
  }
  
  // 処理済みとしてマーク
  processedIds.add(contentId);
  
  // 即座にレスポンスを返す
  res.status(200).json({ status: 'received' });
});

app.listen(3000);
```

```go
package main

import (
    "encoding/json"
    "fmt"
    "log"
    "net/http"
    "sync"
)

var (
    processedIDs = make(map[string]bool)
    mu           sync.RWMutex
)

type WebhookPayload struct {
    ID         string `json:"id"`
    APIVersion string `json:"api_version"`
    Kind       string `json:"kind"`
    Data       struct {
        Action    string                 `json:"action"`
        ID        string                 `json:"id"`
        Timestamp string                 `json:"timestamp"`
        Content   map[string]interface{} `json:"content"`
    } `json:"data"`
}

func webhookHandler(w http.ResponseWriter, r *http.Request) {
    // デリバリーIDを取得
    deliveryID := r.Header.Get("X-BZ-Delivery")
    
    // ペイロードを解析
    var payload WebhookPayload
    if err := json.NewDecoder(r.Body).Decode(&payload); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    
    // 重複チェック
    mu.RLock()
    if processedIDs[payload.ID] {
        mu.RUnlock()
        w.WriteHeader(http.StatusOK)
        json.NewEncoder(w).Encode(map[string]string{"status": "duplicate"})
        return
    }
    mu.RUnlock()
    
    // データを処理
    fmt.Printf("Received %s event for %s\n", payload.Data.Action, payload.Kind)
    fmt.Printf("Data ID: %s\n", payload.Data.ID)
    
    // データ型に応じた処理
    content := payload.Data.Content
    if ticker, ok := content["ticker"].(string); ok {
        fmt.Printf("Ticker: %s\n", ticker)
    }
    
    // 処理済みとしてマーク
    mu.Lock()
    processedIDs[payload.ID] = true
    mu.Unlock()
    
    // 成功レスポンスを返す
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]string{"status": "received"})
}

func main() {
    http.HandleFunc("/webhook", webhookHandler)
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

<div id="common-issues">
  ### よくある問題
</div>

1. **Webhookが届かない**
   * Webhook URL が正しく設定されていることを確認してください
   * エンドポイントが外部からアクセス可能であることを確認してください
   * API key が有効で有効化されていることを確認してください
   * フィルターですべてのデータ型が除外されていないことを確認してください
   * 地域フィルターが想定しているデータと一致していることを確認してください

2. **重複配信**
   * `id` フィールドを使用して冪等性を実装してください
   * エンドポイントの応答時間を確認してください (応答が遅いと再試行が発生する場合があります) 

3. **認証エラー (401-403)**
   * エンドポイントの認証設定を確認してください
   * API key と token を確認してください
   * 注: 認証エラーが発生すると、再試行は直ちに停止します

4. **タイムアウトエラー**
   * エンドポイントが 30 秒以内に応答することを確認してください
   * 必要に応じてデータを非同期で処理してください
   * すぐに成功レスポンスを返し、処理は後で行ってください

5. **想定外の action 形式**
   * レガシーな webhook キーを使用していないか確認してください (action が小文字) 
   * 先頭が大文字の action を受け取るには、新しい webhook キーに更新してください
   * 複数のクライアントをサポートする場合は、両方の形式に対応してください

6. **必要なデータ型が届かない**
   * Webhook 設定に必要なデータ型が含まれていることを確認してください
   * 地域フィルターを確認してください (US/Canada/India の設定) 
   * 日付フィルターによって最近のデータが除外されていないことを確認してください
