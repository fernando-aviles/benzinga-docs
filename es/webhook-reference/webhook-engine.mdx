El motor de webhooks envía actualizaciones de calendarios y señales directamente a tu Punto final HTTP, con reintentos, filtrado y opciones de transformación integrados. Usa esta guía para comprender la mecánica de entrega, la estructura de la carga útil y las prácticas recomendadas de integración.

<div id="overview">
  ## Descripción general
</div>

El servicio Benzinga Data Webhook entrega datos de calendario y señales en tiempo real a los Puntos finales de webhook que configures. Cuando se crean, actualizan o eliminan eventos de calendario (resultados, dividendos, calificaciones, etc.) o señales (actividad de opciones, suspensiones, etc.), el servicio envía automáticamente solicitudes HTTP POST a tu URL de webhook con el payload de datos.

Capacidades clave:

* Alcances configurables para la cobertura de calendarios y señales, para que solo recibas los datos que necesitas
* Entregas idempotentes con un encabezado `X-BZ-Delivery` único y un campo `id` en el payload para la deduplicación
* Programación de reintentos sólida que pasa de reintentos exponenciales rápidos a intentos por hora durante un período prolongado
* Transformaciones de contenido opcionales para adaptar los payloads a los requisitos de los sistemas de destino

<div id="webhook-delivery">
  ## Entrega de webhooks
</div>

<div id="http-request-details">
  ### Detalles de la solicitud HTTP
</div>

* **Method**: `POST`
* **Content-Type**: `application/json`
* **User-Agent**: `Benzinga-Dispatch/v1.0.0 {build-version}`
* **Custom Header**: `X-BZ-Delivery` - Un UUID único para cada intento de entrega (útil para evitar duplicados)

<div id="retry-policy">
  ### Política de reintentos
</div>

El servicio de webhooks de datos implementa un mecanismo de reintento sólido:

* **Fase exponencial**: 15 reintentos durante los primeros 5 minutos
* **Reintentos exponenciales adicionales**: 11 reintentos más, si es necesario
* **Fase de intervalo fijo**: 12 reintentos por hora × 24 horas/día × 7 días (para reintentos prolongados)
* **Tiempo máximo de espera**: 5 minutos entre reintentos en la fase exponencial
* **Tiempo de espera por solicitud**: 30 segundos por solicitud

<div id="response-requirements">
  ### Requisitos de la respuesta
</div>

Su Punto final de webhook debe devolver uno de los siguientes códigos de estado HTTP:

* **Códigos de éxito** (200-202, 204): Indican una entrega exitosa. No se realizarán reintentos.
* **Códigos de error del cliente** (401-403): Indican un fallo de autenticación/autorización. Los reintentos se **detendrán de inmediato** para evitar más intentos fallidos.
* **Otros códigos** (4xx, 5xx): Activarán reintentos según la política de reintentos anterior.

**Importante**: Su Punto final debe responder rápidamente (idealmente en un plazo de 30 segundos) para evitar un tiempo de espera. El motor reintentará en caso de tiempo de espera.

<div id="payload-fields">
  ### Campos del payload
</div>

<div id="top-level-fields">
  #### Campos de nivel superior
</div>

* **`id`** (string, UUID): Identificador único de esta entrega de webhook. Úselo para la deduplicación.
* **`api_version`** (string): Identificador de versión de la API. Actualmente, `"webhook/v1"`.
* **`kind`** (string): Identificador de la ruta del tipo de dato. Consulte [Tipos de datos compatibles](#supported-data-types) para conocer todos los valores posibles.

<div id="data-object">
  #### Objeto de datos
</div>

* **`action`** (string): Tipo de acción del evento. Valores posibles:
  * `"Created"`: Se crearon nuevos datos (valor predeterminado para las nuevas claves de webhook)
  * `"Updated"`: Se actualizaron datos existentes
  * `"Removed"`: Se eliminaron datos
  * **Nota**: Las claves de webhook heredadas pueden recibir valores en minúsculas: `"created"`, `"updated"`, `"removed"`
* **`id`** (string): Identificador único del registro de calendario o señal
* **`timestamp`** (string, ISO 8601): Marca de tiempo en que se generó el webhook
* **`content`** (object): Los datos del calendario o la señal. La estructura varía según el tipo de dato (consulta [Tipos de datos compatibles](#supported-data-types))

<div id="event-actions">
  ## Acciones de evento
</div>

El servicio de webhooks de datos envía eventos para tres tipos de acciones:

1. **Creado**: Se activa cuando se publican nuevos datos de calendario o de señales
2. **Actualizado**: Se activa cuando se modifican datos existentes
3. **Eliminado**: Se activa cuando se eliminan datos

**Nota**: El formato de la acción depende de la configuración de tu webhook:

* **Nuevas claves de webhook**: Reciben acciones con inicial mayúscula (`"Created"`, `"Updated"`, `"Removed"`)
* **Claves de webhook heredadas**: Reciben acciones en minúsculas (`"created"`, `"updated"`, `"removed"`)

<div id="content-filtering">
  ## Filtrado de contenido
</div>

La configuración del webhook puede incluir filtros para controlar qué datos recibes:

<div id="filter-options">
  ### Opciones de filtrado
</div>

* **Tipos de datos**: Filtre por tipos específicos de calendario/señal (p. ej., solo resultados, solo calificaciones)
* **Filtros geográficos**: Controle si quiere recibir:
  * Datos del mercado de EE. UU. (`AllowUSA`)
  * Datos del mercado canadiense (`AllowCanada`)
  * Datos del mercado indio (`AllowIndia`) - para datos de WIIMs
* **Filtro de fecha**: Excluya los datos históricos anteriores a una fecha especificada (`MaxHistoricalDate`)

<div id="exchange-filtering">
  ### Filtrado por bolsa
</div>

El servicio filtra automáticamente por bolsa en función de tu configuración geográfica:

* **Bolsas de EE. UU.**: NYSE, NASDAQ, AMEX, ARCA, OTC, OTCBB, PINX, PINK, BATS, IEX
* **Bolsas canadienses**: TSX, TSXV, CSE, CNSX

<div id="content-transformation">
  ## Transformación de contenido
</div>

El motor de webhooks admite la transformación de contenido para tipos de datos específicos. Las transformaciones se aplican en función de la configuración de tu webhook y pueden incluir:

* Cambio de nombre de campos
* Conversión del formato de los datos
* Filtrado o eliminación de campos

<div id="best-practices">
  ## Mejores prácticas
</div>

<div id="1-idempotency">
  ### 1. Idempotencia
</div>

Usa el campo `id` (UUID) en el payload para implementar la idempotencia. Almacena los ID de entrega ya procesados para evitar procesarlos por duplicado.

<div id="2-response-handling">
  ### 2. Gestión de la respuesta
</div>

* Devuelve `200 OK` o `204 No Content` en cuanto recibas el webhook
* Procesa los datos de forma asíncrona si es necesario
* No realices operaciones prolongadas antes de responder

<div id="3-error-handling">
  ### 3. Manejo de errores
</div>

* Devuelve los códigos de estado HTTP adecuados
* Para errores de autenticación (401-403), asegúrate de que tu Punto final esté configurado correctamente
* Para fallos temporales, devuelve códigos de estado 5xx para activar los reintentos

<div id="4-security">
  ### 4. Seguridad
</div>

* Usa HTTPS para el Punto final de tu webhook
* Implementa autenticación/autorización (claves de API, tokens, etc.)
* Valida el encabezado `X-BZ-Delivery` para mayor seguridad

<div id="5-monitoring">
  ### 5. Monitoreo
</div>

* Supervise los tiempos de respuesta de su Punto final
* Configure alertas ante fallos repetidos
* Realice un seguimiento del encabezado `X-BZ-Delivery` para identificar los intentos de entrega

<div id="6-data-type-handling">
  ### 6. Manejo de tipos de datos
</div>

* Verifica el campo `kind` para determinar el tipo de dato
* Analiza el objeto `content` según la estructura del tipo de dato
* Gestiona distintos formatos de acción (en mayúsculas vs. en minúsculas) si admites claves heredadas

<div id="python-flask">
  ### Python (Flask)
</div>

```python
from flask import Flask, request, jsonify
import uuid

app = Flask(__name__)
processed_ids = set()

@app.route('/webhook', methods=['POST'])
def webhook():
    # Obtener el ID de entrega para deduplicación
    delivery_id = request.headers.get('X-BZ-Delivery')
    
    # Parsear el payload
    payload = request.json
    content_id = payload['id']
    
    # Verificar duplicados
    if content_id in processed_ids:
        return jsonify({'status': 'duplicate'}), 200
    
    # Procesar datos
    action = payload['data']['action']
    kind = payload['kind']
    content = payload['data']['content']
    
    print(f"Received {action} event for {kind}")
    print(f"Data ID: {content.get('id')}")
    
    # Gestionar diferentes tipos de datos
    if 'earnings' in kind:
        print(f"Earnings: {content.get('ticker')} - {content.get('date')}")
    elif 'ratings' in kind:
        print(f"Rating: {content.get('ticker')} - {content.get('rating_current')}")
    elif 'option_activity' in kind:
        print(f"Option Activity: {content.get('ticker')} - {content.get('volume')}")
    
    # Marcar como procesado
    processed_ids.add(content_id)
    
    # Devolver éxito inmediatamente
    return jsonify({'status': 'received'}), 200
```

const express = require('express');
const app = express();
app.use(express.json());

const processedIds = new Set();

app.post('/webhook', (req, res) => {
  // Obtener el ID de entrega para deduplicación
  const deliveryId = req.headers['x-bz-delivery'];

<div id="go">
  ### Go
</div>

```go
package main

import (
    "encoding/json"
    "fmt"
    "log"
    "net/http"
    "sync"
)

var (
    processedIDs = make(map[string]bool)
    mu           sync.RWMutex
)

type WebhookPayload struct {
    ID         string `json:"id"`
    APIVersion string `json:"api_version"`
    Kind       string `json:"kind"`
    Data       struct {
        Action    string                 `json:"action"`
        ID        string                 `json:"id"`
        Timestamp string                 `json:"timestamp"`
        Content   map[string]interface{} `json:"content"`
    } `json:"data"`
}

func webhookHandler(w http.ResponseWriter, r *http.Request) {
    // Obtener ID de entrega
    deliveryID := r.Header.Get("X-BZ-Delivery")
    
    // Parsear payload
    var payload WebhookPayload
    if err := json.NewDecoder(r.Body).Decode(&payload); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    
    // Verificar duplicados
    mu.RLock()
    if processedIDs[payload.ID] {
        mu.RUnlock()
        w.WriteHeader(http.StatusOK)
        json.NewEncoder(w).Encode(map[string]string{"status": "duplicate"})
        return
    }
    mu.RUnlock()
    
    // Procesar datos
    fmt.Printf("Received %s event for %s\n", payload.Data.Action, payload.Kind)
    fmt.Printf("Data ID: %s\n", payload.Data.ID)
    
    // Gestionar diferentes tipos de datos
    content := payload.Data.Content
    if ticker, ok := content["ticker"].(string); ok {
        fmt.Printf("Ticker: %s\n", ticker)
    }
    
    // Marcar como procesado
    mu.Lock()
    processedIDs[payload.ID] = true
    mu.Unlock()
    
    // Retornar éxito
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]string{"status": "received"})
}

func main() {
    http.HandleFunc("/webhook", webhookHandler)
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="common-issues">
  ### Problemas comunes
</div>

1. **No se reciben webhooks**
   * Verifica que la URL de tu webhook esté configurada correctamente
   * Comprueba que tu Punto final sea accesible públicamente
   * Asegúrate de que tu clave de API sea válida y esté activa
   * Verifica que los filtros no estén excluyendo todos los tipos de datos
   * Comprueba que tus filtros geográficos coincidan con los datos que esperas

2. **Entregas duplicadas**
   * Implementa idempotencia usando el campo `id`
   * Comprueba los tiempos de respuesta de tu Punto final (las respuestas lentas pueden provocar reintentos)

3. **Errores de autenticación (401-403)**
   * Verifica la configuración de autenticación de tu Punto final
   * Comprueba las claves de API y los tokens
   * Nota: Los errores de autenticación detienen los reintentos de inmediato

4. **Errores por tiempo de espera**
   * Asegúrate de que tu Punto final responda en un plazo de 30 segundos
   * Procesa los datos de forma asíncrona si es necesario
   * Devuelve una respuesta de éxito de inmediato y procesa los datos después

5. **Formato de acción inesperado**
   * Comprueba si estás usando una clave de webhook antigua (acciones en minúsculas)
   * Actualiza a una nueva clave de webhook para recibir acciones con inicial mayúscula
   * Maneja ambos formatos si das soporte a varios clientes

6. **Faltan tipos de datos**
   * Verifica que la configuración de tu webhook incluya los tipos de datos deseados
   * Comprueba los filtros geográficos (configuración de EE. UU./Canadá/India)
   * Asegúrate de que los filtros de fecha no estén excluyendo datos recientes
