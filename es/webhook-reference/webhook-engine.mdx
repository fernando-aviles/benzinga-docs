El motor de webhooks envía actualizaciones de calendario y señales directamente a su endpoint HTTP, con reintentos integrados, filtrado y opciones de transformación. Use esta guía para comprender la mecánica de entrega, la estructura del payload y las prácticas recomendadas de integración.

<div id="overview">
  ## Descripción general
</div>

El servicio de Webhook de Datos de Benzinga envía datos de calendarios y señales en tiempo real a los endpoints de webhook que hayas configurado. Cuando se crean, actualizan o eliminan eventos del calendario (resultados, dividendos, calificaciones, etc.) o señales (actividad de opciones, suspensiones, etc.), el servicio envía automáticamente solicitudes HTTP POST a la URL de tu webhook con el payload de datos.

Capacidades clave:

* Cobertura configurable para calendarios y señales, de modo que solo recibas los datos que necesitas
* Entregas idempotentes con un encabezado `X-BZ-Delivery` único y el campo `id` del payload para la deduplicación
* Un esquema de reintentos robusto que pasa de reintentos exponenciales rápidos a intentos por hora de larga duración
* Transformaciones de contenido opcionales para adaptar los payloads a los requisitos de los sistemas de destino

<div id="webhook-delivery">
  ## Entrega de webhooks
</div>

<div id="http-request-details">
  ### Detalles de la solicitud HTTP
</div>

* **Método**: `POST`
* **Content-Type**: `application/json`
* **User-Agent**: `Benzinga-Dispatch/v1.0.0 {build-version}`
* **Encabezado personalizado**: `X-BZ-Delivery` - Un UUID único para cada intento de entrega (útil para eliminar duplicados)

<div id="retry-policy">
  ### Política de reintentos
</div>

El servicio de webhook de datos implementa un mecanismo sólido de reintentos:

* **Fase exponencial**: 15 reintentos en los primeros 5 minutos
* **Reintentos exponenciales adicionales**: 11 reintentos más, si es necesario
* **Fase de intervalo fijo**: 12 reintentos por hora × 24 horas/día × 7 días (para reintentos a largo plazo)
* **Tiempo máximo de espera**: 5 minutos entre reintentos en la fase exponencial
* **Tiempo de espera de la solicitud**: 30 segundos por solicitud

<div id="response-requirements">
  ### Requisitos de respuesta
</div>

El endpoint de webhook debe devolver uno de los siguientes códigos de estado HTTP:

* **Códigos de éxito** (200-202, 204): Indican una entrega correcta. No se realizarán reintentos.
* **Códigos de error del cliente** (401-403): Indican un fallo de autenticación/autorización. Los reintentos se **detendrán de inmediato** para evitar más intentos fallidos.
* **Otros códigos** (4xx, 5xx): Activarán reintentos según la política de reintentos anterior.

**Importante**: El endpoint debe responder rápidamente (idealmente en menos de 30 segundos) para evitar un timeout. El motor volverá a intentarlo en caso de timeout.

<div id="payload-fields">
  ### Campos del payload
</div>

<div id="top-level-fields">
  #### Campos de nivel superior
</div>

* **`id`** (string, UUID): Identificador único de esta entrega del webhook. Úselo para eliminar duplicados.
* **`api_version`** (string): Identificador de la versión de la API. Actualmente, `"webhook/v1"`.
* **`kind`** (string): Identificador de la ruta del tipo de dato. Consulte [Tipos de datos compatibles](#supported-data-types) para ver todos los valores posibles.

<div id="data-object">
  #### Objeto de datos
</div>

* **`action`** (string): Tipo de acción del evento. Valores posibles:
  * `"Created"`: Se crearon nuevos datos (valor predeterminado para nuevas claves de webhook)
  * `"Updated"`: Se actualizaron datos existentes
  * `"Removed"`: Se eliminaron datos
  * **Nota**: Las claves de webhook heredadas pueden recibir valores en minúsculas: `"created"`, `"updated"`, `"removed"`
* **`id`** (string): Identificador único del registro de calendario o señal
* **`timestamp`** (string, ISO 8601): Marca de tiempo en la que se generó el webhook
* **`content`** (object): Los datos del calendario o de la señal. La estructura varía según el tipo de dato (consulte [Tipos de datos compatibles](#supported-data-types))

<div id="event-actions">
  ## Acciones de evento
</div>

El servicio de webhook de datos envía eventos para tres tipos de acciones:

1. **Created**: Se activa cuando se publican nuevos datos de calendario o de señales
2. **Updated**: Se activa cuando se modifican datos existentes
3. **Removed**: Se activa cuando se eliminan datos

**Nota**: El formato de `action` depende de la configuración del webhook:

* **Nuevas claves de webhook**: Reciben acciones con inicial mayúscula (`"Created"`, `"Updated"`, `"Removed"`)
* **Claves de webhook heredadas**: Reciben acciones en minúsculas (`"created"`, `"updated"`, `"removed"`)

<div id="content-filtering">
  ## Filtrado de contenido
</div>

La configuración de tu webhook puede incluir filtros para controlar qué datos recibes:

<div id="filter-options">
  ### Opciones de filtrado
</div>

* **Tipos de datos**: Filtre por tipos específicos de calendario o señal (p. ej., solo resultados, solo calificaciones)
* **Filtros geográficos**: Controle si desea recibir:
  * Datos del mercado de EE. UU. (`AllowUSA`)
  * Datos del mercado canadiense (`AllowCanada`)
  * Datos del mercado indio (`AllowIndia`) - para datos de WIIMs
* **Filtro por fecha**: Excluya datos históricos anteriores a una fecha específica (`MaxHistoricalDate`)

<div id="exchange-filtering">
  ### Filtrado por exchange
</div>

El servicio filtra automáticamente por exchange según su configuración geográfica:

* **Exchanges de EE. UU.**: NYSE, NASDAQ, AMEX, ARCA, OTC, OTCBB, PINX, PINK, BATS, IEX
* **Exchanges canadienses**: TSX, TSXV, CSE, CNSX

<div id="content-transformation">
  ## Transformación de contenido
</div>

El motor de webhooks admite la transformación de contenido para tipos de datos específicos. Las transformaciones se aplican en función de la configuración de tu webhook y pueden incluir:

* Cambio de nombre de campos
* Conversión de formatos de datos
* Filtrado/eliminación de campos

<div id="best-practices">
  ## Prácticas recomendadas
</div>

<div id="1-idempotency">
  ### 1. Idempotencia
</div>

Usa el campo `id` (UUID) en el payload para implementar la idempotencia. Almacena los IDs de entrega ya procesados para evitar el procesamiento duplicado.

<div id="2-response-handling">
  ### 2. Manejo de la respuesta
</div>

* Devuelve `200 OK` o `204 No Content` inmediatamente al recibir el webhook
* Procesa los datos de forma asíncrona si es necesario
* No realices operaciones prolongadas antes de responder

<div id="3-error-handling">
  ### 3. Gestión de errores
</div>

* Devuelva los códigos de estado HTTP adecuados
* Para errores de autenticación (401-403), asegúrese de que su endpoint esté configurado correctamente
* Para fallos temporales, devuelva códigos de estado 5xx para activar los reintentos

<div id="4-security">
  ### 4. Seguridad
</div>

* Usa HTTPS para el endpoint de tu webhook
* Implementa autenticación/autorización (claves de API, tokens, etc.)
* Valida el encabezado `X-BZ-Delivery` para mayor seguridad

<div id="5-monitoring">
  ### 5. Monitoreo
</div>

* Supervise los tiempos de respuesta de su endpoint
* Configure alertas para fallos repetidos
* Realice un seguimiento del encabezado `X-BZ-Delivery` para identificar los intentos de entrega

<div id="6-data-type-handling">
  ### 6. Manejo de tipos de datos
</div>

* Verifique el campo `kind` para determinar el tipo de datos
* Analice el objeto `content` según la estructura del tipo de datos
* Maneje distintos formatos de `action` (con mayúscula inicial vs. minúscula) si admite claves heredadas

<div id="python-flask">
  ### Python (Flask)
</div>

```python
from flask import Flask, request, jsonify
import uuid

app = Flask(__name__)
processed_ids = set()

@app.route('/webhook', methods=['POST'])
def webhook():
    # Obtener ID de entrega para deduplicación
    delivery_id = request.headers.get('X-BZ-Delivery')
    
    # Parsear payload
    payload = request.json
    content_id = payload['id']
    
    # Verificar duplicados
    if content_id in processed_ids:
        return jsonify({'status': 'duplicate'}), 200
    
    # Procesar datos
    action = payload['data']['action']
    kind = payload['kind']
    content = payload['data']['content']
    
    print(f"Received {action} event for {kind}")
    print(f"Data ID: {content.get('id')}")
    
    # Gestionar diferentes tipos de datos
    if 'earnings' in kind:
        print(f"Earnings: {content.get('ticker')} - {content.get('date')}")
    elif 'ratings' in kind:
        print(f"Rating: {content.get('ticker')} - {content.get('rating_current')}")
    elif 'option_activity' in kind:
        print(f"Option Activity: {content.get('ticker')} - {content.get('volume')}")
    
    # Marcar como procesado
    processed_ids.add(content_id)
    
    # Devolver respuesta de éxito inmediatamente
    return jsonify({'status': 'received'}), 200
```

const express = require('express');
const app = express();
app.use(express.json());

const processedIds = new Set();

app.post('/webhook', (req, res) => {
  // Obtener ID de entrega para deduplicación
  const deliveryId = req.headers['x-bz-delivery'];
  
  // Parsear payload
  const payload = req.body;
  const contentId = payload.id;
  
  // Verificar duplicados
  if (processedIds.has(contentId)) {
    return res.status(200).json({ status: 'duplicate' });
  }
  
  // Procesar datos
  const { action, content } = payload.data;
  const kind = payload.kind;
  
  console.log(`Received ${action} event for ${kind}`);
  console.log(`Data ID: ${content.id}`);
  
  // Gestionar diferentes tipos de datos

  } else if (kind.includes('option_activity')) {
    console.log(`Option Activity: ${content.ticker} - ${content.volume}`);
  }
  
  // Marcar como procesado
  processedIds.add(contentId);
  
  // Devolver respuesta de éxito inmediatamente

<div id="go">
  ### Go
</div>

```go
package main

import (
    "encoding/json"
    "fmt"
    "log"
    "net/http"
    "sync"
)

var (
    processedIDs = make(map[string]bool)
    mu           sync.RWMutex
)

type WebhookPayload struct {
    ID         string `json:"id"`
    APIVersion string `json:"api_version"`
    Kind       string `json:"kind"`
    Data       struct {
        Action    string                 `json:"action"`
        ID        string                 `json:"id"`
        Timestamp string                 `json:"timestamp"`
        Content   map[string]interface{} `json:"content"`
    } `json:"data"`
}

func webhookHandler(w http.ResponseWriter, r *http.Request) {
    // Obtener ID de entrega
    deliveryID := r.Header.Get("X-BZ-Delivery")
    
    // Parsear payload
    var payload WebhookPayload
    if err := json.NewDecoder(r.Body).Decode(&payload); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    
    // Verificar duplicados
    mu.RLock()
    if processedIDs[payload.ID] {
        mu.RUnlock()
        w.WriteHeader(http.StatusOK)
        json.NewEncoder(w).Encode(map[string]string{"status": "duplicate"})
        return
    }
    mu.RUnlock()
    
    // Procesar datos
    fmt.Printf("Received %s event for %s\n", payload.Data.Action, payload.Kind)
    fmt.Printf("Data ID: %s\n", payload.Data.ID)
    
    // Gestionar diferentes tipos de datos
    content := payload.Data.Content
    if ticker, ok := content["ticker"].(string); ok {
        fmt.Printf("Ticker: %s\n", ticker)
    }
    
    // Marcar como procesado
    mu.Lock()
    processedIDs[payload.ID] = true
    mu.Unlock()
    
    // Retornar éxito
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(map[string]string{"status": "received"})
}

func main() {
    http.HandleFunc("/webhook", webhookHandler)
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="common-issues">
  ### Problemas comunes
</div>

1. **No se reciben webhooks**
   * Verifique que la URL de su webhook esté configurada correctamente
   * Compruebe que su endpoint sea accesible públicamente
   * Asegúrese de que su clave de API sea válida y esté activa
   * Verifique que los filtros no estén excluyendo todos los tipos de datos
   * Compruebe que sus filtros geográficos coincidan con los datos que espera

2. **Entregas duplicadas**
   * Implemente idempotencia usando el campo `id`
   * Compruebe los tiempos de respuesta de su endpoint (las respuestas lentas pueden provocar reintentos)

3. **Errores de autenticación (401-403)**
   * Verifique la configuración de autenticación de su endpoint
   * Compruebe las claves de API y los tokens
   * Nota: Los errores de autenticación detienen los reintentos de inmediato

4. **Errores de tiempo de espera**
   * Asegúrese de que su endpoint responda en un plazo máximo de 30 segundos
   * Procese los datos de forma asíncrona si es necesario
   * Devuelva una respuesta de éxito de inmediato y procese después

5. **Formato inesperado de action**
   * Compruebe si está usando una clave de webhook heredada (actions en minúsculas)
   * Actualice a una nueva clave de webhook para recibir actions con inicial mayúscula
   * Maneje ambos formatos si da soporte a varios clientes

6. **Faltan tipos de datos**
   * Verifique que la configuración de su webhook incluya los tipos de datos deseados
   * Compruebe los filtros geográficos (configuración de US/Canada/India)
   * Asegúrese de que los filtros de fecha no estén excluyendo datos recientes
