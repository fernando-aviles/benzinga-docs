<div id="overview">
  ## Descripción general
</div>

Use este endpoint para desencadenar una entrega de prueba de webhook a su endpoint configurado. Esto le permite verificar que su integración de webhook funcione correctamente antes de recibir datos reales de producción.

<div id="testing-your-integration">
  ## Prueba de la integración
</div>

Cuando llame a este endpoint, Benzinga enviará un payload de prueba del webhook a la URL `destination` configurada. Esta entrega de prueba sigue el mismo formato y la misma lógica de reintento que las entregas de webhooks en producción.

<div id="what-to-expect">
  ### Qué puedes esperar
</div>

1. **Respuesta inmediata**: La API devuelve un código de estado `200` si la entrega de prueba se activó correctamente.
2. **Payload de prueba**: Tu endpoint de webhook recibe un payload de prueba con el mismo formato que los datos de producción.
3. **Encabezados de entrega**: La entrega de prueba incluye el encabezado `X-BZ-Delivery`, igual que las entregas de producción.

<div id="verify-your-integration">
  ### Verifique su integración
</div>

Use este endpoint para confirmar:

* Que su endpoint de webhook sea accesible públicamente
* Que su endpoint pueda procesar correctamente el formato del payload del webhook
* Que su endpoint responda con los códigos de estado adecuados (2xx para indicar éxito)
* Que su endpoint responda dentro del tiempo de espera de 30 segundos
* Que su lógica de idempotencia gestione correctamente el encabezado `X-BZ-Delivery` y el campo `id` del payload

<div id="best-practices">
  ## Mejores prácticas
</div>

* Pruebe primero con endpoints de webhook que no sean de producción
* Verifique que su endpoint responda con los códigos de estado `200` o `204`
* Confirme que sus sistemas de registro y monitoreo capturen la entrega de prueba
* Compruebe que su lógica de deduplicación funcione con el ID de la entrega de prueba
* Pruebe escenarios de error devolviendo temporalmente códigos de estado de error

<div id="troubleshooting">
  ## Solución de problemas
</div>

<div id="424-delivery-error">
  ### 424 Error de entrega
</div>

Si recibe un código de estado `424`, el sistema no pudo entregar la payload de prueba al endpoint de destino. Causas comunes:

* La URL de destino no es accesible públicamente
* El endpoint de destino devuelve códigos de estado de error
* Problemas de conectividad de red
* Errores del certificado SSL/TLS en el endpoint de destino

<div id="400-invalid-request">
  ### 400 Solicitud no válida
</div>

Verifica que se proporcionen todos los parámetros obligatorios y que tengan el formato correcto:

* `destination` debe ser una URL HTTPS válida
* `version` debe ser `webhook/v1`
* `kind` debe ser `News/v1`
